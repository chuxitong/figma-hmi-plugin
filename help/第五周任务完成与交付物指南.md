## 一、任务书第5周到底要你做什么？

用三条「每周交付物」概括（翻译稿与下图一致）：

1. **一个可工作的渲染模块** —— 把 **HTML 字符串** 丢进**无头浏览器**里，**截一张图**（PNG）出来。  
2. **一个最小的 Figma 插件界面** —— 能打开，并且能通过 **HTTP** 和**本机上的服务**说上话。  
3. **一段很短的演示视频或 GIF** —— 本项目在 **`reports/deliverables_week5/week-5-plugin-shell-demo.mp4`**；内容简介见同级 **`DEMO-VIDEO-README.md`**。（若需重新导出，可把新文件替换同名占位。）

面板上**具体要有**哪些按钮、哪些区域，任务书列得很细（约 **L200–208**）：

- 「生成代码」、 「使其更接近设计稿」、 编辑**文字说明**、 「应用编辑」、 **代码输出区**、 **预览**、 **状态/日志**。

**本仓库里**这些都有对应：按钮/区域在插件 **`figma-plugin/src/ui.html`** 里；与任务书**逐条对照**见 **`reports/deliverables_week5/week-5-delivery.md`** 和 **`reports/deliverables_week5/week-5-task-spec-index.md`**。

## 二、和「第4周、第6周」的关系（避免混淆）

- **第4周**做的是：**本地 HTTP 服务** + **API**（如 `/generate`、`/refine`、`/edit`）。  
- **第5周**在本仓库里主要是两件事：  
  - **渲染**：服务里把 **HTML → PNG**（`renderer` + `POST /render`）；  
  - **插件外壳**：Figma 里**一块 UI**，`fetch` 到 `http://localhost:8000`（或你填的 **Service URL**）。  
- **第6周**任务书会强调更完整的**端到端**流程与更多截图交付；**本仓库的 Generate 按钮已经会导出画框再调服务**，这是为了方便联调，**不否定第5周**「外壳 + 渲染」的交付，细节仍以 **`week-5-delivery.md`** 的表述为准。

## 四、常用命令是干什么的（按使用顺序）

以下路径请按你机器上仓库根目录**改成自己的**；示例用 **`/root/autodl-tmp/figma-hmi-plugin`**（常见于 **Linux 云服务器**，例如 AutoDL；若你在 **Windows** 上克隆仓库，请改成如 **`D:\figma-hmi-plugin`** 这类路径）。

### 先分清：命令要在「哪台机器」上执行

本指南里会反复出现两种环境，**不要混在同一台机子上硬套路径**：

| 环境 | 指什么 | 典型场景 |
|------|--------|----------|
| **A. 服务机** | 跑 **Python + uvicorn** 的那台机器；**渲染**（Playwright）也在**同一进程所在机器**上执行 | 云 GPU **服务器**上跑服务；或**你自己电脑**上本地跑 `uvicorn` |
| **B. 用户/开发机** | 装 **Figma 桌面版**、并（通常）在 **Node** 下编译插件 的那台电脑 | 你自己的 **Windows / macOS**；与 A **可以不是同一台** |

- **4.1～4.4、4.3**：只在 **A（服务机）** 上执行（你要 SSH 上服务器，就在**服务器上的终端**里跑；服务若装在自己笔记本，就在**笔记本终端**里跑）。  
- **4.5 `curl … http://127.0.0.1:8000/…`**：必须在与 **「服务监听的 8000 端口」同一台机** 上，把 `127.0.0.1` 当成本机回环时，在 **A 的终端**里执行；**不能**在 B 上访问「服务器上的 127.0.0.1」——在 B 上应把地址改成**隧道/公网可访问的 Service URL**（与插件里填的 **Service URL** 一致），或先在 B 上建 **SSH 端口转发** 再对 `http://127.0.0.1:8000` 发请求。  
- **4.6 `npx tsc`**：在**装了 Node.js、放有本仓库的目录**上执行，实践中几乎总是 **B（Figma/前端开发用电脑）**；**不是**在只有 Python 的无头服务器上跑（服务器若没装 Node，就应在 B 上改完插件、编译、再同步/上传）。  
- **4.7、4.8**：只在 **B（Figma 所在机器）** 上操作/录制。

---

### 4.1 进入本地服务目录

**【执行环境：A 服务机】**在运行/安装 Python 服务的那台电脑的终端里执行（SSH 进云主机后的 Shell 算「服务机上的终端」）。

```bash
cd /root/autodl-tmp/figma-hmi-plugin/local-service
```

**含义**：`cd` = change directory，后面是本仓库里放 **Python 服务** 的文件夹（`app.py`、`renderer.py` 在这）。

### 4.2（仅首次/新环境）安装 Python 依赖

**【执行环境：A 服务机】**与 4.1 相同，在**跑 uvicorn 的那台机子**的终端里执行。

```bash
.venv/bin/pip install -r requirements.txt
```

**含义**：用虚拟环境里的 `pip` 装 **`requirements.txt`** 里列好的包。  
若还没有 `.venv`，需先建虚拟环境再执行（与第四周说明相同，见 **`第四周任务完成与交付物指南.md`** 或根目录 **`README.md`**）。

### 4.3（仅首次）安装无头浏览器内核（给渲染模块用）

**【执行环境：A 服务机】**与 4.1 相同；**渲染在服务端完成**，Playwright 的 Chromium 必须装在**服务进程所在机器**上。

```bash
.venv/bin/playwright install chromium
```

**含义**：`renderer.py` 用 **Playwright** 开 **Chromium** 无头模式截 PNG；这条是**下载浏览器二进制**，不装会报错或无法截图。

### 4.4 启动本地 HTTP 服务

**【执行环境：A 服务机】**在放有 `local-service`、且已建 venv 的终端里执行；保持该窗口**常开**，不要关。

```bash
.venv/bin/python -m uvicorn app:app --host 0.0.0.0 --port 8000
```

**含义**：

- **`uvicorn`**：ASGI 服务器，把 **`app:app`**（`app.py` 里的 `app` 对象）在 **8000 端口**跑起来。  
- **`--host 0.0.0.0`**：不只看本机环回，**同一台机器/端口映射**上别处也能访问；只本机可改为默认或 `127.0.0.1`。

**【在哪打开浏览器看 /docs】**若你在 **A 上**有图形环境或云厂商提供的 **Web 终端+端口映射**，在**能访问到该主机 8000 端口的浏览器**里打开；若服务在**远程**、你人在 **B**，一般用 **B 的浏览器** 访问 `https://你的映射地址` 或经 **端口转发** 后的 `http://127.0.0.1:8000`（以平台文档为准）。  
服务起来后，凡能连到该地址的浏览器，都可打开 **`…/docs`** 看接口文档。

### 4.5 只验证「渲染模块」（不跑大模型）

**【执行环境】**二选一，取决于你能不能让当前终端**访问到正在监听的服务**：

1. **服务跑在 A，且你人就在 A 的终端里（如 SSH 登录到服务器后）** → 在 **A** 上**再开一个**终端（与已跑 `uvicorn` 的窗口分开），执行下面命令；此时 `127.0.0.1:8000` 表示 **A 本机** 上的服务。  
2. **服务在远程 A，你只会操作本机 B** → 下面 **`curl` 里不要把 URL 写死成 `127.0.0.1`**，应改成**你在插件 Service URL 里能访问到的**那个基地址（公网/隧道/转发后的 `http://127.0.0.1:8000`），并在 **B 的终端**（或任意能连到该 URL 的环境）里执行。  
3. **服务与 Figma 都在 B 本机** → 在 **B 的终端**里执行，且 `127.0.0.1:8000` 对。

```bash
curl -sS -X POST "http://127.0.0.1:8000/render" \
  -H "Content-Type: application/json" \
  -d '{"html_code": "<!DOCTYPE html><html><body><h1>Week5</h1></body></html>", "width": 400, "height": 200}'
```

**含义**：

- **`curl`**：命令行发 HTTP 请求。  
- **`-X POST`**：用 POST 方法。  
- **`/render`**：对应 `app.py` 的渲染接口。  
- **`-d '…'`**：JSON 请求体，里面是**最小一段 HTML** 和**视口宽高**。  
- 若成功，返回 JSON 里会有 **`image_base64`**，即 **PNG 的 base64**（证明**无头浏览器已渲染**）。

> **`/render` 不调用大模型**；只证明「第5周渲染模块」工作正常。

### 4.6 编译 Figma 插件（把 TypeScript 编成 Figma 能跑的 JS）

**【执行环境：B 用户/开发机（装 Figma + 通常也装 Node 的那台）】**在**你自己电脑**上打开终端或 PowerShell，进入你克隆下来的仓库里的 **`figma-plugin`** 目录。  
*若你**只在服务器**上工作、本机没有仓库，应先把本仓库**克隆或同步到 B**，或把 `figma-plugin` 拷到 B 再编译——**Figma 读的是 B 上磁盘里的 manifest 与 `dist/`，`npx tsc` 应在有 Node 且被 Figma 引用的那套文件上跑**。*

```bash
cd /root/autodl-tmp/figma-hmi-plugin/figma-plugin
```

*（上例路径是 Linux；Windows 上请用你自己的盘符，例如 `cd D:\figma-hmi-plugin\figma-plugin`。）*

```bash
npx tsc
```

**含义**：

- **`npx tsc`**：用项目里的 **TypeScript 编译器** 按 **`tsconfig.json`** 把 **`src/code.ts`** 等编译到 **`dist/code.js`**。  
- **`manifest.json`** 的 **`main`** 指向 **`dist/code.js`**，不编译 Figma 会读不到新逻辑或报错。

### 4.7 在 Figma 里加载开发插件

**【执行环境：B 用户机（仅 GUI，无 shell 命令）】**在 **Figma 桌面版** 里用鼠标点菜单，**不能**在 Linux 无头服务器上完成。

在 Figma Desktop：**Menu → Plugins → Development → Import plugin from manifest…**，选择**你本机**上克隆仓库里的 **`figma-plugin/manifest.json`**（路径在 **B** 的磁盘上）。

**含义**：Figma 会读 manifest，加载 **`ui.html`** 为面板、**`dist/code.js`** 为沙箱主线程脚本。

### 4.8 录演示 GIF / 视频

**【执行环境：B 用户机】**在装 **Figma**、能录到画面的那台电脑上用 **录屏 / GIF 工具** 操作。  
演示视频：**`reports/deliverables_week5/week-5-plugin-shell-demo.mp4`**（简介见 **`reports/deliverables_week5/DEMO-VIDEO-README.md`**）。

## 五、面板按钮与任务书字面对照（怎么点）

| 任务书（中文） | 面板上英文 / 位置 | 行为简述 |
|----------------|-------------------|----------|
| 生成代码 | **Generate Code** | 主线程把选中 **Frame** 导出 PNG → 调 **`POST /generate`** |
| 使其更接近设计稿 | **Make It Closer to the Mockup** | 用 **`POST /render`** 得当前图 + **`POST /refine`** |
| 编辑指令 + 应用编辑 | **Edit by Request** 输入框 + **Apply Edit** | **`POST /edit`** |
| 代码输出区 | **Generated Code** | 显示返回的 `code` |
| 预览 | **Preview** | 显示 `preview_base64`（PNG） |
| 状态/日志 | **Log** | 请求状态与错误信息 |

**Service URL** 填在 **B（Figma 所在电脑）** 的插件面板上。默认 **`http://localhost:8000`** 表示「**本机**上的服务」——仅当 **uvicorn 就运行在 B 上** 时才成立；若服务在 **远程 A**，这里应填 **A 的公网/隧道/转发地址**。**`manifest.json`** 已用 **`allowedDomains: ["*"]` + `reasoning`** 满足 Figma 对联网权限的校验，一般**不必**为每个新主机再改 manifest；若你改为白名单制，需自行为每个域名建条目（见 Figma 官方 **Plugin manifest → networkAccess**）。

## 六、若 Figma 报 Manifest error：networkAccess / localhost

若底部出现类似红色提示：

- *Invalid value for networkAccess. If you want to allow localhost, please add a "reasoning" field … or use "devAllowedDomains" …*

说明 **Figma 新规**：在 **`allowedDomains` 里只写 `http://localhost:8000` 而不带 **`reasoning`** 会不通过。本仓库**已**在 **`figma-plugin/manifest.json`** 中改为 **`allowedDomains: ["*"]` + 必填的 `reasoning` 字段**（对论文/内测插件足够，且可访问你在面板里填的任意 `Service URL`）。

**你需要做的**：

1. 拉取/同步最新 **`figma-plugin/manifest.json`**。  
2. 在 **B 机** 于 **`figma-plugin`** 目录再执行一次 **`npx tsc`**（只改了 manifest 也可以不 tsc，但建议养成习惯）。  
3. 在 Figma：**Plugins → Development →** 对你已导入的插件点 **Remove** 或 **重新 Import plugin from manifest…** 本仓库的 **`figma-plugin/manifest.json`**，让 Figma 重新读 manifest。  
4. 再打开插件试一次。

若仍失败，在 Figma **Help** 或开发文档中核对当前桌面版对 `networkAccess` 的字段要求（随版本可能微调）。

**仍同一句红字时，十之八九是：Figma 读到的不是仓库里最新 `manifest`（本机有旧复制 / 用 zip 老目录 / 未重新 Import）**。请按 **`figma-plugin/MANIFEST-TROUBLESHOOTING.zh.md`** 逐步核对「记事本打开的路径 = Import 的路径」并 Remove 插件后重导。
