# 毕业论文任务计划（中文翻译）

## 论文题目

**俄语：** Разработка интеллектуального программного модуля генерации и итеративной корректировки кода человеко-машинных интерфейсов по графическим макетам

**英语：** Development of an Intelligent Software Module for the Generation and Iterative Refinement of Human-Machine Interface Code from Graphical Mockups

**中文：** 基于图形化设计稿的人机界面代码智能生成与迭代修正软件模块开发

---

## 项目概述

目标是开发一个智能 Figma 插件原型，用于工业人机界面（HMI）的设计，例如操作面板、监控仪表盘、报警画面、设备状态页面和监控控制界面。该系统不应被呈现为一键完美转换器。它的目的是帮助用户获得工业界面的初始代码版本，迭代改进该版本使其更接近参考设计稿，并支持通过自然语言指令进行有针对性的编辑。

此工作流程的技术基础是现实可行的。Figma 插件可以使用 `exportAsync` 将选中的节点或画框导出为编码图像，通过 `figma.variables` 访问设计变量，通过 `getCSSAsync` 获取 Inspect 风格的 CSS 提示，并通过 `figma.ui.postMessage` 和 `parent.postMessage` 在插件逻辑和插件用户界面之间交换数据。

视觉语言模型 UI2Code^N 是一个合适的基线模型，因为其论文和公开仓库将其描述为一个交互式开放模型，支持 UI 到代码的生成、UI 编辑和 UI 优化，并提供了公开的演示脚本。

该系统应被呈现为工业界面开发的智能工程助手，而非完全自主的完美设计到代码解决方案。

**其主要实用价值在于：**

- 减少实现早期阶段的手工工作量
- 生成有用的界面代码初始版本
- 支持根据参考设计稿进行迭代优化
- 允许通过自然语言请求进行有针对性的修改
- 展示如何将 AI 集成到工业 HMI 工程工作流程中

---

## 插件目标功能

插件应在 Figma 内提供一个简单的面板，包含三个主要功能。

### 功能一：生成代码

用户选择一个画框并点击按钮。系统导出选中的画框，发送到本地 AI 服务，接收生成的 HTML/CSS 代码，在插件窗口中显示该代码，展示渲染结果的小预览图，并提供复制生成代码的操作。

### 功能二：使其更接近设计稿

系统将从 Figma 导出的原始截图与从生成代码渲染的截图进行比较。然后要求模型改进代码，使渲染结果更接近参考图像。这种优化可重复 1-3 次迭代。

### 功能三：按指令编辑

用户用自然语言编写指令，例如：

- "把这个按钮改成次要样式"
- "增大标题字号"
- "减少趋势图上方的间距"
- "使报警区块更醒目"

系统根据此请求修改当前代码并显示更新后的预览。

这三个功能与 UI2Code^N 描述的交互模式完全对应：生成、编辑和优化。

## 计划

### 每周提交的通用格式

第1周应创建一个 GitHub 或 GitLab 仓库，并将链接发送给导师一次。在后续所有周中，你应该发送：

- 同一个更新后的仓库链接
- 简短的周报（`reports/week-N.md` 或 PDF）
- 截图或短演示视频
- 关于当前问题和下一步的简要说明

每周必须以一个具体且可验证的成果结束。导师每周五莫斯科时间 20:00 进行检查。

---

### 第1周：研究 Figma 工作流程并搭建项目仓库

研究 Figma 插件如何与选中的文档交互以及可以从中提取哪些数据。至少必须理解如何：

- 获取当前选中的画框
- 检查选中内容及其结构
- 将画框导出为 PNG
- 访问设计变量
- 获取界面元素的 CSS 提示
- 在插件逻辑和插件 UI 之间交换消息

**每周交付物：**

- 包含初始项目结构和 README 的仓库链接
- 一份简短的技术说明，描述哪些 Figma 数据是必须的、哪些是可选的、以及如何将这些数据传递给外部服务
- 一个最小可行验证，例如：导出选中画框为 PNG，或在插件逻辑与插件 UI 之间发送测试消息

---

### 第2周：准备工业 HMI 模型集

准备一个小而有代表性的工业风格界面模型集，将在整个项目中用于测试和演示。

**推荐策略：**

- 取 2-3 个可编辑的 Figma 仪表盘模板
- 取 2-4 个工业 HMI/SCADA 视觉参考
- 在 Figma 中创建 5-8 个改编的工业模型

**推荐的可编辑 Figma 来源（基础布局）：**

- Figma Dashboard Templates
- Themesberg Figma Admin Dashboard Template
- Sneat Free Figma Admin Dashboard UI Kit

**推荐的工业视觉参考：**

- Siemens HMI Template Suite
- Ignition Public Demonstration
- AggreGate SCADA/HMI Gallery
- FUXA
- JointJS SCADA/HMI Demo

**模型集建议组成：**

- 2 个简单画面
- 2 个中等复杂度画面
- 1-2 个较难画面

**建议的画面类型：**

- 设备状态仪表盘
- 报警和事件画面
- 趋势监控画面
- 操作控制面板
- 生产线概览
- 设备状态卡片

**每周交付物：**

- 包含 5-8 个工业风格画框的 Figma 文件或共享链接
- 这些画框导出的 PNG 图片存入仓库
- 一个简短的描述表格：画面名称、预期工业用途、复杂度等级、入选理由

---

### 第3周：启动并验证基线模型

在本地环境中安装并运行基线模型 UI2Code^N。必须验证该模型能否：

- 从 UI 截图生成代码
- 根据文字指令编辑代码
- 使用迭代视觉反馈改进代码

**每周交付物：**

- 安装笔记和环境描述
- 2-3 个基线测试示例
- 一份简短报告，描述基线模型结果的主要优势和典型错误
- 基线输出的截图

---

### 第4周：将基线模型封装为本地服务

将模型封装为一个小型本地 HTTP 服务。Figma 插件不能在内部运行模型。相反，插件应向外部进程发送请求。

**服务应接受：**

- 界面的 PNG 图片
- 任务描述或提示词（prompt）
- （可选）当前代码
- （可选）来自 Figma 的变量
- （可选）CSS 提示

**服务应返回：**

- 生成或更新的 HTML/CSS 代码
- （可选）简短说明
- （可选）下一阶段的元数据

服务可以使用以下任一方式：

- 独立端点，如 `/generate`、`/refine`、`/edit` 和 `/render`
- 一个统一端点，带有明确定义的操作模式

**每周交付物：**

- 仓库中可运行的本地服务代码
- 简短的 API 描述，包含请求和响应示例
- 报告中包含一个演示请求和一个演示响应

---

### 第5周：添加代码渲染并构建首个插件外壳

实现一个渲染模块，将生成的 HTML/CSS 转换为图像。

**该模块应：**

- 接受代码作为输入
- 在无头浏览器中打开
- 渲染页面
- 生成截图

然后构建第一个可工作的 Figma 插件外壳，包含：

- 一个"生成代码"按钮
- 一个"使其更接近设计稿"按钮
- 一个编辑指令的文本框
- 一个"应用编辑"按钮
- 一个代码输出区
- 一个预览区
- 一个小的状态/日志区

**每周交付物：**

- 一个可工作的渲染模块
- 一个最小的 Figma 插件面板，可打开并与本地服务通信
- 一个简短的演示视频或 GIF，展示外壳的工作情况

---

### 第6周：实现代码生成和基本优化

实现第一个完整的端到端工作流程。

**第一部分：生成代码**

当用户按下"生成代码"时，插件应：

- 验证已选中一个画框
- 将画框导出为 PNG
- （可选）收集变量和 CSS 提示
- 将数据发送到本地服务
- 接收 HTML/CSS
- 显示代码
- 渲染并展示预览
- 允许用户复制代码

**第二部分：基本优化**

实现"使其更接近设计稿"的第一个版本：

- 存储来自 Figma 的原始 PNG
- 将当前代码渲染为截图
- 将参考图像、渲染图像和当前代码发送到服务
- 请求代码优化
- 显示更新后的结果

**每周交付物：**

- 在第2周的至少 3 个工业模型上演示工作的生成模式
- 在至少 2 个模型上演示工作的优化模式
- 优化前后的结果截图

---

### 第7周：实现按指令编辑，并添加可选的变量和 CSS 提示

实现对当前代码的自然语言编辑。用户应能提交如下指令：

- "把按钮改成次要样式"
- "增大标题字号"
- "减少左侧内边距"
- "使报警区块更醒目"

同时添加开关，允许用户包含或排除：

- Figma 设计变量
- 选中元素的 CSS 提示

**每周交付物：**

- 一个可工作的"按指令编辑"模式
- 可工作的变量和 CSS 提示开关
- 至少 3 个编辑示例及截图
- 至少 1 个小对比：仅图像输入 vs. 图像加变量和 CSS 提示

---

### 第8周：对原型进行实验评估

进行结构化的实验评估。

**至少测量以下指标：**

- 获得首次生成结果所需的时间
- 获得可接受结果所需的时间
- 优化迭代次数
- 典型错误类别
- 启用变量的效果
- 启用 CSS 提示的效果

**同时描述常见的失败模式，例如：**

- 间距不正确
- 字号不正确
- 对齐不佳
- 视觉层次错误
- 按钮样式不准确
- 元素缺失
- 图表、卡片或趋势区域再现性差

**每周交付物：**

- 实验部分的草稿，包含表格和图表
- 评估流程的清晰描述
- 一页简短的结论，总结主要观察结果

---

### 第9周：改进基线并冻结原型

在基线原型工作并评估完成后，尽可能改进它。可能的方向：

- 更好的提示词工程
- 更清晰的请求格式
- 更好地利用变量和 CSS 提示
- 生成代码的后处理
- 改进的优化逻辑
- 针对工业界面示例的小型适配

到本周结束时，系统应被视为功能完整，以便第10周可以专注于写作。

**每周交付物：**

- 基线版本与改进版本的对比
- 仓库中代码的最终版本
- 一份报告，说明改进了什么以及是否有帮助
- 原型的标签发布或冻结提交

---

### 第10周：文献综述和毕业论文写作

本周应专注于论文写作，而非添加新的主要功能。

**你需要：**

- 准备一份简洁的文献综述
- 按照格式要求组织论文
- 撰写并排版完整的论文草稿
- 将准备好的论文草稿发送给导师审阅

对于文献综述，合理的做法是从 UI2Code^N 的相关工作部分开始，然后加入与截图到代码生成和界面数据集相关的论文和数据集，例如：Design2Code、WebSight、Web2Code

**每周交付物：**

- 将完整的论文文本发送给导师检查
