本文记录本次针对 **Figma 插件报错 / DevTools Issues / 渲染异常 / manifest 校验** 的一次完整排查。目标是区分：

1. **真正需要修的项目问题**  
2. **Figma 自己页面、Chromium 或第三方资源产生的噪声**

并把**我改了哪些代码、为什么这样改、你该如何验证**写清楚。

---

## 一、先说结论：你截图里的报错，不是每一条都属于“插件代码错了”

结合你发来的截图、项目代码、以及 Figma 官方文档，本次看到的现象可分成三类：

### A. 需要认真修的“真问题”

1. **`manifest.json` 的 `networkAccess` 校验不过**
   - 这是**真问题**，会直接导致 Figma 不加载插件。
   - 典型红字：`Invalid value for networkAccess ... localhost ... reasoning ... devAllowedDomains ...`

2. **插件前端对服务响应处理过于乐观**
   - 之前 UI 里多处 `fetch(...).then(resp.json())`，**没先判断 `resp.ok`**。
   - 一旦服务返回 **4xx / 5xx / 非 JSON**（例如 HTML 错页），插件会报一串“JSON parse”类错误，但真正原因被遮住。

3. **预览区用 `innerHTML` 直接拼接 `preview_base64`**
   - 虽然服务端理论上应该只返回 PNG 的 base64，但直接拼 HTML 字符串不够稳妥。
   - 更安全的做法是先验证 base64，再用 DOM API 创建 `<img>`。

4. **渲染器对缺少 `<!DOCTYPE html>` 的 HTML 没做兜底**
   - 模型有时返回的是 HTML 片段，或完整文档但**没有 doctype**。
   - Chromium 会因此进入 **Quirks Mode（怪异模式）**，布局、间距、字号都可能变形。
   - 这会表现成“渲染结果怪”“跟浏览器里看不一样”。

5. **渲染器首次并发请求时，可能重复启动多个 Playwright 浏览器**
   - 原来 `_get_browser()` 没有并发保护。
   - 多个请求同时进来时，理论上可能重复初始化浏览器实例。

### B. 多数情况下可忽略的“噪声”

1. **`Deprecated feature used: Unload event listeners are deprecated`**
   - 这类通常来自 **Figma 自己加载的资源** 或其嵌入的 Chromium 环境。
   - 截图里资源名像 `vendor-core-...`，**不是我们项目里的脚本文件**。
   - 这不是我们插件逻辑层面的主故障。

2. **`Ignored @property rule` / `inherits`**
   - 截图里受影响资源是 `figma_app-...min.css.br`
   - 这明显是 **Figma 自身 CSS**，不是我们的 `ui.html` 或服务端渲染内容。
   - 可视为 **Figma 页面自己的 CSS 兼容性提醒**。

### C. 半真半假的“现象型问题”

1. **Quirks Mode**
   - 截图里出现 `data:text/html;base64,...` 的 Quirks Mode 提示。
   - 这类提示**不一定**直接来自插件面板本身，但它暴露出一个很重要的风险：  
     **如果任何被嵌入/渲染的 HTML 缺少 doctype，Chromium 的确会进怪异模式。**
   - 所以这条提示虽然**未必精确指向我们的文件**，但我们仍应在渲染器里做兜底修复。

## 二、我实际查了什么资料

本次排查不是“拍脑袋猜”，而是对照了：

1. **Figma 官方文档：Plugin Manifest**
   - 结论：
     - `allowedDomains` 如果包含 `*`，**必须**有 `reasoning`
     - 若使用 `localhost` / development server，官方更推荐补 `devAllowedDomains`
     - `documentAccess: "dynamic-page"` 是当前文档中的推荐配置

2. **Figma 官方文档：Making Network Requests**
   - 结论：
     - 插件的联网白名单由 `networkAccess` 决定
     - 开发期访问本机地址，`devAllowedDomains` 是推荐手段

3. **Figma Forum / 相关问题**
   - 结论：
     - 很多人看到的 `networkAccess` 报错，根因其实是：
       - manifest 写法不符合新规则
       - 或 **Figma 仍在读取旧 manifest**

4. **浏览器/Chromium 常识**
   - 缺 `<!DOCTYPE html>` 确实会导致 **Quirks Mode**
   - 渲染链路里若 HTML 来源不稳定，应在服务端统一规范化

### 2. `figma-plugin/src/ui.html`

### 问题 1：服务响应没先检查状态码

以前的写法是：请求回来后直接 `resp.json()`。  
如果服务返回：

- 404
- 500
- HTML 错误页
- 反向代理返回的一段纯文本错误

插件只会抛出模糊的 JSON 解析错误，**真正的 HTTP 错误被掩盖**。

### 修复

新增了统一的 `readServiceJson()`：

```150:188:figma-plugin/src/ui.html
function setPreviewImage(previewBase64) {
  // ...
}

async function readServiceJson(resp, opName) {
  const text = await resp.text();
  if (!resp.ok) {
    const snippet = text.slice(0, 400).replace(/\s+/g, " ").trim();
    throw new Error(`${opName} failed: HTTP ${resp.status} ${resp.statusText}${snippet ? " — " + snippet : ""}`);
  }
  try {
    return text ? JSON.parse(text) : {};
  } catch (err) {
    const snippet = text.slice(0, 400).replace(/\s+/g, " ").trim();
    throw new Error(`${opName} returned non-JSON content${snippet ? " — " + snippet : ""}`);
  }
}
```

现在 `/generate`、`/refine`、`/edit`、`/render` 都改为先走这个函数。

### 效果

- 出错时你能在插件的 **Log** 区看到类似：
  - `POST /render failed: HTTP 500 Internal Server Error — ...`
  - `POST /generate returned non-JSON content — <html>...`

而不是一堆“莫名其妙的 JSON 报错”。

### 问题 2：预览用 `innerHTML` 拼接

以前是：

```html
previewArea.innerHTML = `<img src="data:image/png;base64,${result.preview_base64}" ...>`
```

这会带来两类问题：

1. 如果服务返回的不是干净 base64，会把 DOM 搞乱
2. 调试时不容易分辨是“预览无效”还是“DOM 注入失败”

### 修复

我改成了：

- 先做 base64 形态检查
- 再用 `document.createElement("img")`
- 再设置 `img.src`

对应代码见：

```156:174:figma-plugin/src/ui.html
function setPreviewImage(previewBase64) {
  const area = document.getElementById("previewArea");
  if (!area) return;
  area.textContent = "";
  if (!previewBase64 || typeof previewBase64 !== "string") {
    area.textContent = "No preview yet";
    return;
  }
  const cleaned = previewBase64.replace(/\s+/g, "");
  if (!/^[A-Za-z0-9+/=]+$/.test(cleaned)) {
    area.textContent = "Preview rejected: invalid base64.";
    log("Preview rejected: service returned a non-base64 preview payload.");
    return;
  }
  const img = document.createElement("img");
  img.alt = "preview";
  img.src = "data:image/png;base64," + cleaned;
  area.appendChild(img);
}
```

### 效果

- 预览异常时会直接在面板里提示 **invalid base64**
- 不再使用拼接式 `innerHTML`

### 3. `local-service/renderer.py`

### 问题 1：首次并发请求可能重复启动浏览器

原先 `_get_browser()` 没有锁：

- 请求 A 进来，发现 `_browser is None`
- 请求 B 也同时进来，也发现 `_browser is None`

理论上会导致重复初始化。

### 修复

新增了 `asyncio.Lock()`：

```13:36:local-service/renderer.py
_browser = None
_playwright = None
_browser_lock = asyncio.Lock()

async def _get_browser():
    global _browser, _playwright
    if _browser is not None:
        return _browser
    async with _browser_lock:
        if _browser is None:
            from playwright.async_api import async_playwright
            _playwright = await async_playwright().start()
            _browser = await _playwright.chromium.launch(headless=True)
    return _browser
```

### 效果

- 首次高并发时更稳
- Playwright 不会被重复拉起

---

### 问题 2：HTML 缺 doctype 会触发 Quirks Mode

这是本次排查里最值得修的“隐性问题”之一。  
即便截图里的 Quirks Mode 不一定 100% 来自我们的插件，它暴露出的风险是真实的：

- 模型可能返回 `<html>...</html>` 但没有 `<!DOCTYPE html>`
- Chromium 进入怪异模式
- PNG 截图布局错位

### 修复

新增 `_normalise_html_document()`，若没有 doctype 就自动补上：

```39:52:local-service/renderer.py
def _normalise_html_document(html_code: str) -> str:
    """Ensure the browser receives a standards-mode HTML document."""
    text = (html_code or "").lstrip("\ufeff")
    stripped = text.lstrip()
    if not stripped.lower().startswith("<!doctype html"):
        logger.info("render_html_to_png: prepending missing <!DOCTYPE html>.")
        return "<!DOCTYPE html>\n" + text
    return text
```

并在真正写临时文件前统一调用：

```119:127:local-service/renderer.py
    browser = await _get_browser()
    page = await browser.new_page(viewport={"width": width, "height": height})
    goto_ms = _goto_timeout_ms()
    page.set_default_timeout(goto_ms)
    html_doc = _normalise_html_document(html_code)

    tmp = tempfile.NamedTemporaryFile(suffix=".html", delete=False, mode="w", encoding="utf-8")
    try:
        tmp.write(html_doc)
```

### 效果

- 即使模型只吐出 fragment，也会被尽量拉回标准模式
- 减少截图和浏览器直接查看时的布局偏差

## 九、本次追加排查（2026-04-26 晚）：`localhost:8765` 与 `catch {}` 语法错误

你后续贴出的新日志里，出现了两组**高信号**关键词：

1. **`localhost:8765/... ERR_CONNECTION_REFUSED`**  
2. **`Syntax error on line 28: Unexpected token { catch {`**

这两组问题的性质**完全不同**。

### 9.1 `localhost:8765` / `[bootstrap] UI fetch failed` 不是主插件 `figma-plugin`

这组日志对应的是仓库里的另一个工具插件：

- `_localtools/figma-bootstrap/manifest.json`
- `_localtools/figma-bootstrap/ui.html`
- `_localtools/figma-bootstrap/code.js`

其中源码直接写明了它依赖本地 8765 端口：

```15:21:_localtools/figma-bootstrap/ui.html
<div class="row">Источник: http://localhost:8765 (12 symbols + 8 screens)</div>
// ...
const BASE = "http://localhost:8765";
```

而它的主线程也会把失败明确打印成：

```347:350:_localtools/figma-bootstrap/code.js
if (msg.type === "fetch-error") {
  toast("UI fetch failed: " + msg.text, { error: true, timeout: 8000 });
```

所以，当你看到：

- `localhost:8765/symbols/pump.svg`
- `[bootstrap] UI fetch failed`

正确解释是：

- 你运行/导入的**不是只有主插件**
- 还有一个 **HMI Auto Bootstrap (local)** 工具插件在尝试访问本地素材服务
- 你本机**没有启动**那个监听 `localhost:8765` 的素材服务，所以它报 `ERR_CONNECTION_REFUSED`

这**不是** `figma-plugin` 主插件自己的 `/generate`、`/render`、`/edit` 逻辑报错。

### 9.2 `catch {}` 是主插件真实兼容性问题，已经修复

你日志里的这条：

```text
Syntax error on line 28: Unexpected token {
    catch {
          ^
```

命中了主插件编译产物 `figma-plugin/dist/code.js` 的旧写法。  
此前 `figma-plugin/src/code.ts` 里用了两处 **optional catch binding**：

- `catch { ... }`

按现代 JS 标准这合法，但 Figma 当前这套嵌入解析环境对它不稳，所以会直接把主线程脚本卡死。

我已把它们改成显式写法：

```16:30:figma-plugin/dist/code.js
    try {
        const css = await node.getCSSAsync();
        // ...
    }
    catch (_err) {
        // getCSSAsync may not be available in all API versions
    }
```

以及（**历史说明**：较早的 `dist/code.js` **仅**调用 `getLocalVariablesAsync`；当前版本见 `src/code.ts` **`collectVariables`**——子树 `boundVariables` + 库变量绑定）：

```35:66:figma-plugin/dist/code.js
    try {
        const localVars = await figma.variables.getLocalVariablesAsync();
        // ...
    }
    catch (_err) {
        // Variables API may not be available
    }
```

并已重新编译：

- `cd figma-plugin && npx tsc`

现在新的 `dist/code.js` **不再包含** `catch {}`。

### 9.3 浏览器能打开 `http://127.0.0.1:8000/docs`，但 Figma 里点 Generate 仍「完全没反应」

这两件事**不矛盾**：

- 在 Chrome **地址栏直接打开** `http://127.0.0.1:8000` 时，页面与 API **同源**，不会走「公网页 → 本机 127.0.0.1」那一套 CORS 专用网络预检。
- Figma 插件的 UI 跑在**独立的安全上下文**里，用 `fetch("http://127.0.0.1:8000/...")` 时，Chromium 会发带 `Access-Control-Request-Private-Network` 的 **OPTIONS 预检**；若**实际响应该预检的 HTTP 服务**（你若是 **SSH 转发**，则是隧道另一侧云主机上的 `uvicorn`；若是本机直跑，则是本机 `uvicorn`）未在 CORS 里返回 `Access-Control-Allow-Private-Network: true`，预检失败，**真正的 POST 根本不会发出**，面板上便既没有代码也没有明确报错（或只在 DevTools 里看到被拦截）。

**代码修复**（`local-service/app.py`）：在 `CORSMiddleware` 上设置 **`allow_private_network=True`**（Starlette 会据此在预检里返回 `Access-Control-Allow-Private-Network: true`）。

**你需要做的**：把最新 `app.py` 部署到**正在跑 `uvicorn` 的那台机器**（云服务器或本机，与你实际一致），并在**该机器上**重启 `uvicorn`。若使用 **AutoDL + SSH 转发**（`ssh -L 8000:127.0.0.1:8000`），Figma 里 **Service URL 仍用** `http://127.0.0.1:8000` 即可，**但** CORS 修复必须发生在**远端的** `app.py` 上。然后再在 Figma 里试 **Generate Code**；面板 **Log** 在点击后应先出现 *Generate: UI click ok…* 或 *Sandbox: exporting…*，然后出现 *Frame "…" exported* 等后续行（见 **§9.5** 若仍只有 `Ready.`）。

### 9.4 你本地现在该怎么做

如果你要验证主插件 `figma-plugin`：

1. 先把**本机** `figma-plugin/` 整个目录同步为最新版本  
2. 重新在 Figma 里 **Remove** 旧开发插件  
3. 再 `Import plugin from manifest...` 指向**本机最新的** `figma-plugin/manifest.json`

### 9.5 点了 Generate 但 Log 仍只有 `Ready.`（联网可查到的两类原因）

对照 [Creating a User Interface](https://developers.figma.com/docs/plugins/creating-ui/) 与社区贴（如 [target origin / 缺少 `'*'`](https://stackoverflow.com/questions/79016571/figma-plugin-development-error-failed-to-execute-postmessage-on-domwindow)、[onmessage 被覆盖](https://stackoverflow.com/questions/61140151/figma-react-plugin-postmessage-not-working-as-expected)）：

1. **`parent.postMessage` 必须带第二参数**，常见为 `'*'`，否则会出现 *target origin … null* 或与 Figma 主机不一致，**主线程收不到消息**。本仓库已改为优先 `https://www.figma.com`、失败再回退 `'*'`，并附带 `pluginId`（与 `manifest.json` 的 `id` 一致），与官方「非 null origin iframe」说明对齐。  
2. **HTML5 中 `<button>` 未写 `type` 时默认为 `submit`**，在插件 iframe 里可能产生非预期行为；已显式 **`type="button"`** 于 Generate / Refine / Edit / Copy。  
3. **`window.onmessage = …` 会覆盖其他监听**；已改为 **`window.addEventListener("message", …)`** 接收主线程回传。  

同步 **`src/ui.html`** 后需重新 **Import** 开发插件；若见 **Build: 2026-04-28** 或更新（如 **2026-05-07** 剪贴板修复）行即表示新 UI 已加载。

### 9.6 点了 「Copy last API request」无痕式失败 / 剪贴板仍为空

**原因**：Figma 插件 UI 跑在 iframe 里，时常 **没有可用的 `navigator.clipboard`**（`undefined`）；旧代码对该 API 直呼 `writeText` 会 **同步抛错**，连 **Log** 里的 catch 都打不到，表现为「点了没动静」。

**代码行为**（`figma-plugin/src/ui.html`）：

1. 若存在 Clipboard API → `writeText`；失败则用隐藏 `<textarea>` + **`document.execCommand('copy')`** 再试。  
2. 仍失败则在 **Log** 打出说明，并把焦点放到 **`Last API request snapshot`** 只读框，请 **Ctrl+A / Ctrl+C**（内容与 Copy 按钮目标一致）。

**自检**：跑一次 **Generate** 后先看 Log 是否有 *Frame … exported*；快照框应出现 JSON（或为 `no_request_yet` 提示）。再点 Copy——**Log** 至少多一行说明（成功时会标出 `Clipboard API` 或 `execCommand …`）。

### 9.7 勾了 Include design variables，`variables` 仍是 `{}`，但画布上 Fill 明明绑了 Libraries 变量

**原因**：旧版沙盒 **只调用** `getLocalVariablesAsync()`，枚举的是「**当前文件里定义的本地变量**」。从 **Libraries** 点到 Fill 上时，多数是 **远端变量别名绑定**，文件中未必有同名本地定义，枚举结果为空，payload 就出现 **`variables: {}`**——属于**插件采集策略不完整**，不是你的稿「没变量」。

**修复**：`src/code.ts`（build **`2026-05-07-r5-vars-bound-tree`** 及之后）改为：对导出根 Frame **DFS**，从 `boundVariables`、每层 paint/stroke/effect/文本段等收集 **`VARIABLE_ALIAS`**，再 **`getVariableByIdAsync` + `resolveForConsumer`** 解析取值（库变量可被解析）。

**你需要做的**：同步 **`figma-plugin/src/code.ts` + 重新 `npx tsc` 后的 `dist/code.js`**（或直接使用仓库产物），Development 插件 **Remove → 再 Import**；沙盒载入后应有一条 notify：`HMI plugin build … r5-vars-bound-tree loaded`。
