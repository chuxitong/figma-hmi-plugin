# Глава 1. Обзор литературы, цель и задачи

## Введение

Работа посвящена программному модулю, который помогает инженеру перевести макет промышленного HMI из Figma в рабочий HTML/CSS-код. Речь не идёт об «идеальной однокнопочной конвертации». Речь о том, чтобы получить начальную версию кода, подтянуть её под референс с помощью нескольких итераций и внести точечные правки короткой фразой на естественном языке. Для промышленных контекстов — операторские панели, дашборды мониторинга, экраны аварий, синоптика — этот сценарий даёт ощутимый выигрыш по сравнению с ручным переписыванием макета по образцу.

Почему тема для меня живая, а не абстрактная. Во-первых, появились VLM, которые тянут UI-to-code и правки по картинке — в том числе UI2Code^N от zai-org (2025), её можно взять готовой. Во-вторых, в Figma Plugin API уже есть экспорт фрейма, переменные, CSS-подсказки и обмен с iframe — то есть данные для сервиса достать реально. В-третьих, промышленный HMI в статьях про screenshot-to-code почти не фигурирует, нормальных открытых датасетов под заводские экраны нет, а руками переносить макет в код всё равно долго — хоть какая-то помощь на старте уже полезна.

## Обзор литературы

Опорная модель — UI2Code^N Янг и соавторов (2025). Это 9-миллиардная визуально-языковая модель на базе GLM-4.1V-9B-Base, дообученная под три задачи: генерация кода по скриншоту, редактирование существующего кода и итеративное «полирование» под референс. Модель выложена на Hugging Face (`zai-org/UI2Code_N`) с открытой лицензией MIT, в карточке есть пример инференса через `transformers`. Я взял её как baseline, потому что три режима совпадают с тем, что нужно в плагине, и дообучать свою модель я не планировал. Минус один — размер: в bf16 она не влезает в 6 GB VRAM моего ноутбука, приходится либо 4-битная квантизация, либо часть слоёв на CPU через accelerate. Отсюда и медленный инференс, зато пайплайн проверить можно.

Рядом с UI2Code^N полезно смотреть на Design2Code (Си и соавторы, 2024). Это не модель, а бенчмарк с собственным тестовым набором, на котором принято сравнивать системы screenshot-to-code. В моей задаче промышленный HMI слишком специфичен, чтобы мериться на Design2Code напрямую, но сама методика оценки — структурированное сопоставление референса и сгенерированной страницы — вдохновляет мои собственные эксперименты.

WebSight (Laurençon и соавторы, 2024) и Web2Code (Юэ и соавторы, 2024) — это большие публичные датасеты пар «скриншот — HTML/CSS», собранные автоматически по веб-страницам. Для моей работы они играют роль методологического контекста: они показывают, какую «плотность примеров» считают необходимой современные методы, и подсказывают, что для полноценного прогресса по промышленным HMI нужен аналогичный размеченный корпус. Его создание выходит за рамки текущей ВКР, но идея зафиксирована как направление дальнейшей работы.

Отдельно стоит упомянуть инженерный контекст. Документация Figma Plugin API прямо определяет, что мы можем вытащить из макета: `exportAsync` для растрового экспорта, `figma.variables` для локальных переменных дизайна, `getCSSAsync` для подсказок в стиле Inspect, `postMessage` для двустороннего обмена плагин/UI. Я пользуюсь всем этим в своём плагине. Для промышленной части я смотрел на публичные шаблоны Siemens HMI Template Suite, открытое демо Ignition SCADA, свободно распространяемую FUXA и галерею JointJS SCADA/HMI Demo. Они не дают кода, но дают типовые визуальные решения: цветовые коды состояний, компактные таблицы аварий, карточные макеты статуса, синоптические схемы. Именно это я и закладывал в свои собственные мокапы (неделя 2), чтобы модель тестировалась на экранах, похожих на продакшен, а не на общих дашбордах админки.

## Цель работы

Цель состоит в разработке прототипа интеллектуального программного модуля, встроенного в Figma, который по графическому макету промышленного HMI выдаёт рабочий HTML/CSS-код, позволяет итеративно приближать результат к референс-макету и принимает точечные правки на естественном языке. Модуль опирается на открытую модель UI2Code^N и строится как распределённая система: плагин в Figma отвечает за взаимодействие с пользователем и данные из макета, а внешний локальный HTTP-сервис — за инференс и рендер.

Практическая ценность состоит не в «полной автоматизации», а в сокращении времени на получение начальной работающей версии экрана и в систематизации правок по циклу «сгенерируй → подправь → скорректируй по тексту». Предполагаемый пользователь — инженер, который проектирует промышленный интерфейс в Figma и хочет быстрее перейти от макета к проверяемой в браузере странице.

## Задачи

Первая задача — изучить возможности Figma Plugin API и сформировать список данных, которые могут быть извлечены из документа и переданы внешнему сервису. Сюда входит экспорт фреймов, извлечение переменных, сбор CSS-подсказок и организация обмена сообщениями между песочницей и UI плагина.

Вторая задача — собрать представительный набор промышленных мокапов (около восьми экранов разной сложности), опираясь на публичные шаблоны и визуальные референсы SCADA/HMI. Этот набор используется во всех экспериментах и документирован отдельной таблицей в `mockups/mockup-index.md`.

Третья задача — запустить базовую модель UI2Code^N в локальном окружении и подтвердить, что она умеет все три нужных режима (генерация, правка, итеративное уточнение). С учётом ограничения по GPU на ноутбуке — подобрать практичную конфигурацию загрузки (4-битная квантизация, автоматический device_map).

Четвёртая задача — оформить модель как локальный HTTP-сервис с понятными эндпоинтами, чтобы плагин не выполнял тяжёлые вычисления внутри себя.

Пятая задача — реализовать модуль рендера HTML/CSS в PNG (headless-браузер) и собрать рабочую оболочку плагина с тремя действиями (Generate Code, Make It Closer to the Mockup, Edit by Request) и двумя опциональными переключателями контекста.

Шестая задача — реализовать и проверить полный сквозной сценарий Generate Code, а также первую версию итеративного уточнения.

Седьмая задача — довести Edit by Request до рабочего состояния и сделать структурированное сравнение трёх уровней контекста: только картинка, картинка плюс переменные Figma, картинка плюс переменные плюс CSS-подсказки.

Восьмая задача — провести воспроизводимую экспериментальную оценку по всем мокапам: тайминги, число итераций до приемлемого результата, распределение типичных ошибок, эффект контекста.

Девятая задача — внести улучшения на основании экспериментальных данных (промпт-преамбула, пост-обработка кода, ограничение итераций) и сравнить baseline с улучшенной версией.

Десятая задача — подготовить текст ВКР и литературный обзор.

## Объект и предмет исследования

Объектом является процесс разработки промышленных HMI по графическим макетам Figma с использованием интеллектуальных моделей генерации кода. Предметом — архитектура и функциональность программного модуля, связывающего Figma, локальный сервис и визуально-языковую модель в сценарии «сгенерируй → уточни → правь словами».

## Методы и инструменты

Baseline-модель — UI2Code^N из репозитория `zai-org/UI2Code_N`, загружается через `transformers` в 4-битной квантизации. Плагин Figma — на TypeScript, сборка `npx tsc`. Локальный сервис — FastAPI, эндпоинты `/generate`, `/refine`, `/edit`, `/render`. Рендер — Playwright с headless-Chromium. Тестовый набор — собственные восемь мокапов в Figma с PNG-экспортом. Для быстрой отладки пайплайна без загрузки тяжёлой модели — детерминированный заместитель с тем же интерфейсом.

## Научная и практическая значимость

Научная часть сосредоточена на систематизации инженерного сценария применения открытой VLM к промышленному HMI и на структурированной оценке эффекта контекстной подачи (переменные дизайна, CSS-подсказки) на качество генерации. Практическая — работающий прототип, встроенный в Figma, с воспроизводимыми артефактами экспериментов и понятной точкой расширения (замена модели через `model_wrapper.py`).

## Структура работы

Далее в работе идёт вторая глава с формализацией задачи и описанием функциональности модуля (без кода), третья глава с результатами экспериментов, заключение, список источников и приложения.
